# 找矿资料系统优化方案

## 背景

系统主流程已跑通：上传 ROI → 解析 → 构造定位 → 知识库查询 → 地质/物探/化探/遥感数据采集 → 报告生成。但在**可靠性、可用性、可维护性**三方面有明显短板：

- 服务重启后所有任务丢失（内存字典存储）
- 网络失败无重试，175MB 物探数据下载一次失败就整步跳过
- 前端单文件 370 行，无法管理历史任务、无地图展示
- 77 处 print() 替代日志，问题排查靠肉眼
- 零测试覆盖，每次改动需手动跑完整流水线验证

---

## 第一阶段：快速加固（1-2 周）

> 不动主流程，纯增量改进，每项可独立上线

### 1.1 配置集中化

**问题**：URL、超时、文件大小限制硬编码在各模块中
**方案**：新建 `prospector/config.py`，集中管理所有常量

```
config.py 内容：
- EMAG2_URL, WGM2012_URL, STAC_URL 等外部数据源地址
- DEFAULT_TIMEOUT, MAX_RETRIES, CHUNK_SIZE
- MAX_UPLOAD_SIZE, DEFAULT_BUFFER_KM
- OUTPUT_DIR, UPLOAD_DIR 路径
```

**涉及文件**：`web_app.py`, `geophy_fetcher.py`, `geochem_fetcher.py`, `live_fetcher.py`, `rs_fetcher.py`, `geo_fetcher.py`

### 1.2 日志系统替代 print()

**问题**：77 处 print() 无法分级、无法持久化、无法追踪
**方案**：新建 `prospector/src/logger.py`，统一日志框架

```
logger.py:
- get_logger(name) 工厂函数
- 输出到 logs/prospector.log + 控制台
- 按 INFO/WARNING/ERROR 分级
```

逐文件替换 print → logger.info/warning/error，保留 emoji 前缀

**涉及文件**：所有 src/ 下的 .py 文件

### 1.3 HTTP 客户端 + 重试机制

**问题**：5 处 requests.get/post 各自硬编码超时，无重试；175MB 下载网络波动直接失败
**方案**：新建 `prospector/src/http_client.py`

```python
# 核心功能：
- 封装 requests.Session
- 429/502/503/504 自动重试（指数退避，最多 3 次）
- 统一超时管理（从 config 读取）
- 流式下载 + 进度回调（从 geophy_fetcher._download_file 提取）
- DEBUG 级别记录请求 URL、状态码、耗时
```

**迁移清单**：
- `geophy_fetcher.py` 的 `_download_file`（流式下载 175MB EMAG2）
- `geochem_fetcher.py` 的 GEOROC API 查询
- `live_fetcher.py` 的 OpenAlex + Semantic Scholar API
- `report_generator.py` 的 MyMemory 翻译 API

### 1.4 任务持久化

**问题**：`tasks: dict = {}` 纯内存存储，重启即丢失，27 个历史任务不可见
**方案**：

- 每次任务状态变更时，同步写入 `output/{project_name}/task_meta.json`
- 服务启动时扫描 OUTPUT_DIR，自动恢复历史任务到 tasks 字典
- 新增 `GET /api/tasks` 端点，返回所有任务列表

```python
# task_meta.json 结构
{
  "id": "abc123",
  "status": "completed",
  "step": "完成",
  "output_dir": "output/xxx",
  "output_name": "xxx",
  "mineral": "铜",
  "created_at": "2026-06-02T22:47:00",
  "result": { ... }
}
```

**涉及文件**：`web_app.py`

### 1.5 过期数据清理

**问题**：output/ 目录无限增长，无清理机制
**方案**：

- 新增 `DELETE /api/tasks/<task_id>` 端点，删除指定任务及其 output 目录
- 启动时扫描超过 N 天（可配置）的目录，打印磁盘用量警告

**涉及文件**：`web_app.py`

## 第二阶段：架构改进（2-4 周）

### 2.1 前端模块化拆分

**问题**：`templates/index.html` 单文件 370 行，CSS/JS/HTML 混杂
**方案**：

```
拆分为：
static/css/app.css     ← 样式
static/js/upload.js    ← 文件上传、拖放、校验
static/js/pipeline.js  ← 提交、轮询、进度
static/js/results.js   ← 结果展示、报告查看、下载
templates/index.html   ← 纯 HTML 骨架
```

引入轻量状态对象替代全局变量：
```javascript
const AppState = {
  tasks: {},
  currentTaskId: null,
  pollTimers: {},
};
```

### 2.2 历史任务管理界面

**问题**：前端只能看当前任务，无法查看/复用历史结果
**方案**：

- 页面加载时调用 `GET /api/tasks` 获取历史列表
- 添加可折叠的"历史任务"面板，展示：矿种、ROI 文件名、状态、时间
- 点击历史任务可直接查看报告和下载成果包

**依赖**：1.4 任务持久化 + 2.1 前端模块化

### 2.3 异常体系统一

**问题**：各 fetcher 用 try/except Exception 捕获一切，返回 None 静默失败
**方案**：新建 `prospector/src/exceptions.py`

```python
class FetchError(Exception): pass
class NetworkError(FetchError): pass    # 网络超时、连接失败
class DataFormatError(FetchError): pass  # 数据解析异常
class ROIError(Exception): pass         # ROI 文件格式问题
```

各 fetcher 抛出具体异常 → `run_pipeline_thread` 捕获并记录：哪步失败、是否有部分结果、是否建议重试

### 2.4 后台任务队列

**问题**：用 threading.Thread 直接跑流水线，Flask 线程安全隐患 + 无并发限制
**方案**（按优先级）：

- **方案 A**（推荐，无需新依赖）：用 `concurrent.futures.ThreadPoolExecutor` 替代裸线程，限制 max_workers=2
- **方案 B**（如需扩展）：引入 Redis + RQ 任务队列

**涉及文件**：`web_app.py`

### 2.5 基础测试套件

**方案**：添加 pytest，覆盖核心模块

```
tests/
  test_roi_parser.py     ← KML/XLSX 解析、坐标检测、bbox 扩展
  test_mineral_kb.py     ← 8 种矿种查询、别名归一化
  test_tectonic_units.py ← 构造单元定位（已知坐标 → 预期单元）
  test_web_api.py        ← Flask test_client 测试各 API 端点
```

## 第三阶段：体验升级（4-8 周）

### 3.1 地图可视化（Leaflet.js）

**价值**：地质工作者需要空间直观，目前全是文字表格

- ROI 上传后：在地图上显示多边形 + 缓冲区 + 中心点
- 流水线完成后：叠加构造单元边界、含油气盆地轮廓
- 报告中：嵌入内联地图，标注数据源位置

**新增端点**：
- `POST /api/parse-roi` — 仅解析 ROI 返回 GeoJSON（不启动流水线）
- `GET /api/tectonic-overlay?bbox=...` — 返回构造单元 GeoJSON

### 3.2 数据可视化（Chart.js）

**价值**：化探背景值、异常阈值目前只有数字，图表更直观

- 元素背景值柱状图（区域 vs 全国，标注异常阈值带）
- 物探地图内联展示（emag2_map.png、wgm_map.png）
- 论文列表卡片化（引用数徽章、年份标签、可展开摘要）
- Sentinel-2 缩略图网格

### 3.3 细粒度进度反馈

**问题**："收集地球物理资料" 一步要 5+ 分钟，用户看不到子进度
**方案**：

```python
task['sub_steps'] = [
    {"name": "EMAG2 下载", "status": "running", "progress": 45},
    {"name": "WGM2012 下载", "status": "pending"},
]
```

- 各 fetcher 更新 sub_steps
- 前端渲染多步骤进度条（每步独立进度）
- 可选：SSE 替代轮询，减少延迟

### 3.4 项目管理

**问题**：27 个 output 目录平铺，无组织
**方案**：

- 新增"项目"概念：名称 + 描述 + 关联任务列表
- `projects.json` 存储项目元数据
- 前端新增项目选择器和项目管理页

## 关键设计决策

1. **暂不引入数据库** — 阶段一用 per-directory JSON 文件即可，后续可平滑迁移到 SQLite
2. **暂不引入前端框架** — 单页工具用原生 JS + 模块拆分足够，React/Vue 是过度工程
3. **CLI 和 Web 保持对等** — `prospector.py` 和 `web_app.py` 共用同一套 src/ 模块，所有改动保持两端兼容
4. **领域数据不动** — mineral_kb.py、tectonic_units.py 中的地质专业数据不随重构改动

## 验证方式

每个阶段完成后：
1. 用 `tests/test_roi.kml`（铜陵铜矿区）跑完整流水线，确认结果不变
2. 重启 Flask 服务后，通过 `GET /api/tasks` 确认历史任务恢复
3. 检查 `logs/prospector.log` 确认日志输出正常
4. 前端完整走一遍上传 → 进度 → 查看 → 下载流程

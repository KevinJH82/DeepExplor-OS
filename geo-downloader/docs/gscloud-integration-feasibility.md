# gscloud.cn(地理空间数据云)接入本系统 — 可行性调研

> 调研日期:2026-06-03。目标:评估能否把 gscloud.cn 的数据(尤其高分一号 GF-1)程序化接进
> geo-downloader 现有下载/拼接/裁剪/交付流程。结论:**技术可行但代价高、稳定性差、合规存疑;
> 不推荐做登录爬虫,推荐"人工下载 + 系统侧预处理接入"。**

## 一、一句话结论(go / no-go)

- **不是"接个 API"的事**。gscloud 无官方开放 API,只能走 **Selenium 浏览器自动化 + 图形验证码 OCR + 下单等待** 的爬虫路子——脆、不稳、随网页改版即失效,且爬登录态有 ToS 风险。
- **数据价值落差大**:免费的只有 **GF-1 WFV 16m**(比已接入的 Sentinel-2 10m 还粗,对本案 6.82km² 小区块意义有限);**PMS 2m/8m 要付费/签保密协议**,且要下单等待(可达 ~15 天)。
- **建议**:把 gscloud 当"人工下载源",系统只做下游**正射+大气校正+裁剪+交付**;另评估 PIE-Engine / AI Earth 作为更稳的程序化 GF 源。

## 二、技术现状(实测调研)

1. **无官方开放 API。** 检索全网与官网导航,未见公开 RESTful 接口/开发者文档;平台是**动态 JS 站**,检索结果靠 DOM 渲染,不暴露干净 JSON 接口(`requests` 直接打不到真实数据)。
2. **登录 = 账号 + 密码 + 图形验证码。** 自动化必须用 OCR(如 `ddddocr`)破 4 位图形验证码,成功率不稳定、需重试。
3. **检索/下载流程:**
   - 主流公开方案**全是 Selenium + ChromeDriver 模拟点击**(知乎/CSDN 多篇),没有可复用的后台接口。
   - 免费数据(Landsat / DEM)可直接下载,老方法也支持 FTP 批量。
   - **GF / 订单类数据要"加入购物车 → 下单 → 等待"**,非即时直链,等待最长可达约 15 天。
4. **反爬:** 弱,靠 `time.sleep()` 节流即可;但页面结构一变脚本就废(维护成本高)。

## 三、GF-1 数据现状(gscloud)

| 产品 | 分辨率 | 获取 |
|---|---|---|
| GF-1 **WFV** 多光谱 | 16m | **免费**(gscloud 有挂"高分一号WFV数据产品") |
| GF-1 **PMS** 多光谱 | 8m | **付费** |
| GF-1 **PMS** 全色 | 2m | 不免费:科研需申请/签保密协议,商业购买约 **440 元/景起** |

- **数据级别 L1A**:仅辐射校正,**未正射、未大气校正**。进本系统前必须先 **正射(RPC + DEM)→ 大气校正(到地表反射率)**,否则几何对不齐、辐射尺度对不上,无法进拼接/RRN/裁剪,也喂不了下游蚀变分析(后者吃 BOA 反射率)。

## 四、接入本系统的两条路线 + 评估

### 路线 A:写 gscloud Selenium 下载器(mirror `downloader/base.py`)
- **做法**:新增 `downloader/gscloud.py`,Selenium 起 headless Chrome → OCR 破验证码登录 → 模拟检索 → WFV 直下 / 订单类走异步等待(类比 PRISMA/EnMAP 的 `_async_pending_loop` daemon 轮询补包)。
- **代价**:引入 `selenium + chromedriver + ddddocr` 重依赖(与现有"纯 HTTP API"架构格格不入);**脆**(改版即挂);登录态爬取 **ToS 风险**;订单类要等数天。
- **收益**:能半自动化拿 **WFV 16m**(价值有限);PMS 仍要人工付费/申请,自动化救不了。
- **评价**:投入产出比低,不推荐作为首选。

### 路线 B(推荐):不做爬虫,系统只做"接入 + 预处理"
- **做法**:gscloud 的不稳环节(登录/下单/下载)留给人工;新增 `postprocess` 侧接入:**正射(RPC+DEM)+ 大气校正 + 按 KML 裁剪 + 按现有交付结构出件**,复用刚做的 RRN / nodata / 裁剪逻辑。
- **代价**:正射+大气校正需引入处理库(如 GDAL RPC warp + 简易 DOS/6S);一次性工程量中等。
- **收益**:把有价值的预处理沉淀进系统,且对**任何来源的 GF-1**(gscloud / PIE / 商业)都通用;不背爬虫维护与合规债。
- **评价**:稳、通用、合规,**推荐**。

## 五、更稳的程序化 GF 替代源(顺带)

- **PIE-Engine(航天宏图)/ AI Earth(阿里达摩院)**:有 GF 系列数据 + Python SDK,程序化友好,但需对应账号/额度。比爬 gscloud 稳得多,值得评估。
- **CRESDA(资源卫星应用中心)**:权威源但无开放 API,仍是人工门户。
- **若目的只是"这块地要高分辨率"**:已接入的 **Sentinel-2 10m(免费)** 多数场景够用;真要 2m 级,优先商业源(本系统已支持 SPOT/Pleiades,需账号)或 PIE/AI Earth,而非爬 gscloud。

## 六、建议下一步

1. **首选**:走路线 B —— 我先写 GF-1 的"接入+预处理"模块(正射+大气校正+裁剪+交付),你从 gscloud 人工下 WFV(或 PMS)后丢进来即可。
2. **并行评估**:我去核 PIE-Engine / AI Earth 的 GF-1 程序化接口与账号门槛,若可行则它才是真正"自动化 GF"的正道。
3. **仅在你坚持**要 gscloud 全自动时:先做一个**验证码 OCR + 登录 PoC**,实测成功率与稳定性,再决定是否投入路线 A。

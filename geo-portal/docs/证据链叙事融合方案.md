# 证据链叙事:从"自说自话"改为"与本项目真实数据融合"

## Context
用户反馈"证据链叙事"整体格局/逻辑没问题(#1),但**内容与本项目真实结果没融合、还是自说自话**(#2),并给了范式样例 `津巴布韦Gaika金矿…报告.docx`(#3)。
Gaika 范式的灵魂:**每句话挂真实数(品位/坐标/埋深/面积/速率/koz/置信)+ 证据相互印证(地-空-钻三位一体:192孔落入AI异常区→品位+71%、AI预测埋深vs实测误差<10%)+ 因果链(背景→证据综合→靶区由哪些证据交汇支撑→布孔→价值/风险)**。

诊断:`lib/geologyNarrative.js` 把证据降维成状态词("已完成/已降级"),正文套**矿种静态模板**(`MODELS[mineralKey]`),而真实数据(`model3d.targets` 分数/深度、各证据权重/覆盖、`drill` 见矿、`evidencePlan` 权重/rationale、`run.plan.decision_rationale`)**已加载却没用**。
**关键发现**:5 个证据服务**早已在 manifest/metadata 里输出量化指标**,只是 BFF 没转发:
- 蚀变 `anomaly_ratio`、`high_confidence_pixels`;构造 `n_lineaments`/`total_lineament_length_km`/`lineament_density_mean`/`dominant_strikes_deg`;物探 euler `n_sources`/`source_depth_km_*`/`inclination_deg`;化探 `n_anomalies`/`median_ca_log_ratio`/`key_elements`;形变 `deformation_rate_abs_*_mm_yr`/`coverage_ratio`/`n_bursts`。

**已确认方案**:两层生成 = **①确定性事实层(无LLM,即时)** + **②LLM综合研判层(复用 geo-reporter synthesis,按 run 缓存)**;并**把各证据服务已有的 summary 指标接通**到叙事。

## 实施方案(分两期,可独立验收)

### 阶段一:确定性"事实层"——把真实数接通并落到叙事(无 LLM)

**A1. BFF 转发各证据 summary 指标**(`backend/app/main.py`)
- 加 `_evidence_summary(service, meta) -> dict`:从各服务的 manifest/metadata 读已有指标,归一成 `{metrics:{...带单位}, layers_used?}`。映射(字段已存在,见 Context):
  - analyser(`_run_analyser` 读 manifest 处 ~617):`anomaly_ratio`、`high_confidence_pixels`、`deposit_type`。
  - stru(`_run_stru` 读 metadata ~670):`n_lineaments`、`total_lineament_length_km`、`lineament_density_mean`、`dominant_strikes_deg`。
  - geophys(`_geophys_layer_rel`/`_run_geophys` model_stats ~890):euler `n_sources`、`source_depth_km_min/max`、`inclination_deg`/`declination_deg`、`mineral_type`。
  - insar(`_find_insar_evidence*` 的 `insar_metadata.json:stats` ~1009):`deformation_rate_abs_mean/max/p95_mm_yr`、`coverage_ratio`、`n_bursts`。
  - geochem(走通用 svcstatus,非 adapter):在通用分支读其 `metadata.json` 的 `anomaly_stats`(`n_anomalies`/`n_background`/`median_ca_log_ratio`)、`key_elements`。
- adapter 服务把 `summary` 挂到任务 `t["summary"]`(随 `_adapter_thread` 持久化到 adapter_tasks,重启可恢复);`svcstatus` 响应增加 `summary` 字段(adapter 从 `t`,geochem 从 metadata 现读)。

**A2. 前端捕获并持久化 summary**(`frontend/src/store.js`)
- 证据轮询(`runOneEvidenceReal`/批量 poll)读 `s.summary` → `setEvidence(key,{summary})`;`persistEvidenceTask` 带上 summary;`open()`/hydrate 时回灌(从 evidence_plan/adapter 恢复),保证刷新后仍在。

**A3. 重写叙事事实层**(`frontend/src/lib/geologyNarrative.js`)
- `buildProjectNarrative` 各段改为**消费真实数据**,每句挂数;并体现**证据相互印证**:
  - **context**:项目名/矿种/AOI 实际面积/`run.plan.decision_rationale.family_determination`(编排器真实判定)。
  - **logic(成矿逻辑)**:`evidencePlan.model`+`rationale`(真实)+ 本模型要求的证据与权重(`evidence_tasks[].weight/required_level`)+ `datacolle.best_model`/`fit_score`/`pathfinder_elements`。
  - **chain(证据链)**——核心:逐条用 `evidence.summary` 写真实数(如"构造:识别 N 条线性构造、总长 X km、主走向 NW;蚀变:异常占比 Y%、高置信像元 Z");**交叉印证**用 `model3d.stats.available_layers`/`data_sources`(哪些证据真正喂进三维融合)+ 各证据 weight 组合说明靶点由谁支撑;门控 `evidencePlan.gate`(已完成 N/需 M)。
  - **target(靶区)**:`model3d.targets` 前三的 score/depth/坐标 + 由哪些证据层叠合(权重)→ "首位靶点 score 0.87 @ 250m,蚀变30%+构造26%+物探20% 叠合"。
  - **next(布孔/下一步)**:`drill.holes` 孔数/深度/斜孔 + `feedback` 见矿;无 model3d 则"先补 X 证据闭合门控"。
  - **risk**:真实 failed/degraded/缺失证据、低 `coverage_ratio`、单一证据依赖、门控未过。
- 保留现有 6 段结构与 `EvidenceStoryline` 渲染框架(用户说格局没问题);仅换"内容来源"。

### 阶段二:LLM"综合研判层"——复用 geo-reporter synthesis(Gaika 式措辞)

**B1. geo-reporter 暴露轻量 synthesis 接口**(`geo-reporter`)
- `geo-reporter/reporter/synthesis.py:evaluate_synthesis()` 已把真实数组(证据摘要、`deep_targets`、钻孔、`dc_metallogenic`、known_deposits)喂 Claude 产出带交叉印证的研判 + 逐靶点定级。新增 `POST /api/synthesize`:入参=组装好的 run 真实数据,出参=`{grade, summary, dimensions[], target_assessment[]}`(复用现有 prompt/parse,不跑全报告)。

**B2. BFF 综合研判端点 + 缓存**(`backend/app/main.py`)
- `GET /api/runs/{trace}/synthesis`(+ `POST .../synthesis/refresh`):汇集本 run 真实数据(各 `evidence.summary`、`model3d.targets`、`drill`、`datacolle`、project/矿种模型),调 geo-reporter `/api/synthesize`,结果**按 run 缓存**(存 run 记录或 adapter_tasks,键含证据完成指纹);失败/未配置 LLM 时返回 `null`(前端只显事实层,不报错)。

**B3. 前端研判卡**(`store.js` + `EvidenceStoryline.jsx` + `api/portal.js`)
- `api`:`getSynthesis(traceId)`/`refreshSynthesis(traceId)`;`store`:`synthesis` 状态 + `loadSynthesis`(进证据/3D 阶段时拉)。
- `EvidenceStoryline` 增"证据链综合研判(AI)"区:展示 `grade`+`summary`+逐维度(构造/蚀变/物探/化探/形变 的 level+evidence 文字)+逐靶点定级,带"↻ 刷新研判";顶部仍显事实层(A3)。

## 复用(非新写)
各服务 manifest/metadata **已有的量化指标**(无需改服务算法,只接通)、`geo-reporter/synthesis.py` 现成的 LLM 研判 prompt/parse、现有 `model3d.targets`/`drill`/`evidencePlan`/`datacolle`/`run.plan.decision_rationale` 数据、`EvidenceStoryline` 6 段结构与样式、adapter_tasks 持久化、svcstatus 轮询通道。

## 改动文件清单
**后端**:`app/main.py`(`_evidence_summary` + 4 adapter 挂 summary + geochem 通用分支 + svcstatus 带 summary + synthesis 端点/缓存)。
**geo-reporter**:`reporter/synthesis.py`/app 暴露 `POST /api/synthesize`。
**前端**:`lib/geologyNarrative.js`(事实层重写)、`store.js`(summary 捕获/持久 + synthesis 状态)、`components/EvidenceStoryline.jsx`(事实层渲染 + 研判卡)、`api/portal.js`(synthesis API)、必要时 `styles.css`。

## 验证(端到端)
nohup 重启 BFF + reporter + 前端;token 同前。
1. **summary 接通**:对一个有产物的 run,各证据完成后 `svcstatus` 响应含 `summary`(真实数:构造 n_lineaments、蚀变 anomaly_ratio、物探 n_sources、形变 mm/yr、化探 n_anomalies)。
2. **事实层**:证据链叙事每段出现本 run 真实数(靶点 score/depth、证据权重、门控 N/M、钻孔见矿),不再是矿种通稿;换不同项目内容随之变化(非模板)。
3. **交叉印证**:chain/target 段说明"靶点由哪些证据层叠合(权重)"、门控达成情况,体现融合而非罗列。
4. **研判层**:`GET /api/runs/{trace}/synthesis` 返回 grade+逐维度+逐靶点(引用真实分数/深度);storyline 研判卡渲染;刷新可重算;未配 LLM → 事实层照常、研判区显"未启用"。
5. **持久化/回归**:刷新页面叙事与 summary 仍在;原证据生成/3D/报告流程不受影响;`npm run build` 通过。

## 注意/边界
- 不改各证据服务的**算法**,只**接通其已有指标**;若个别指标缺失(如某服务未算 coverage)则该句优雅省略,不编造。
- 研判层是 LLM 产物 → 必须**喂真实数据**(事实层同源)避免幻觉;按 run 缓存,证据指纹变才重算。
- geochem 当前非 adapter(走通用 svcstatus / 经 reporter),summary 从其 metadata 现读;如后续要 adapter 化另议。
- 阶段一可独立上线(无 LLM 依赖,先消灭"自说自话");阶段二在其上叠加 Gaika 式研判。

# 前端设计、实用性与证据链体系优化方案

## 1. 总体判断

当前前端已经具备一套清晰的“勘探指挥中心”雏形：项目空间、编排单、数据准备、证据层、3D 建模、AI 布孔和报告阶段都已经串起来，`trace_id` 也作为主线贯穿流程。

但从真实地质勘探找矿系统的角度看，目前界面更偏“流程演示 + 结果展示”，还没有形成足够严谨的证据链体系。对找矿系统来说，用户不只需要看到“结果是什么”，还必须能追问“这个结果为什么成立、由哪些证据支撑、哪些证据缺失或冲突、是否可复核”。

因此后续前端优化应分成两条主线：

- 设计与实用性：提高布局稳定性、操作效率、状态表达和真实数据可信度。
- 证据链体系：让每个阶段、每个证据层、每个靶点和最终报告都可追溯、可解释、可审计。

## 2. 设计与实用性优化

### 2.1 工作台布局

当前工作台大量使用 `fixed` 定位，包括顶部栏、右侧 HUD、左侧证据栈、底部 dock、中央工作面板、深度滑条等。桌面大屏下视觉效果较好，但小屏、低高度屏幕或浏览器缩放后容易出现遮挡。

建议：

- 将工作台整理为稳定布局：顶部栏、主画布、右侧 Inspector、底部流程栏。
- 右侧 HUD 在窄屏下改为抽屉或底部 sheet。
- 底部流程 dock 增加横向滚动或紧凑模式。
- 中央面板增加最大高度和内部滚动，避免内容超出视口。
- 为常见断点增加验证：`1440x900`、`1280x720`、`1024x768`、移动窄屏。

### 2.2 地图与 3D 表达

当前 2D 地图主要是网格背景和 AOI 框，3D 是 CSS 伪三维体。作为 MVP 演示可接受，但对地质、遥感和矿产预测用户来说，缺少足够的专业上下文。

建议：

- 2D 地图增加比例尺、坐标提示、AOI 面积、bbox、当前图层名称。
- 证据图层增加色带图例，说明颜色代表的指标含义。
- 3D 靶点增加评分色带、深度范围、剖面层含义说明。
- 明确区分真实产物、演示产物、fallback/mock 产物。
- 中期再考虑接入真实地图底图、瓦片、Cesium/Resium 或专业剖面视图。

### 2.3 流程文案与操作语义

当前“生成并执行”实际行为是生成编排并进入数据准备，并不会自动跑完整 pipeline，容易造成误解。

建议：

- 将“生成并执行”改为“生成编排单并进入数据准备”。
- 如果保留“一键执行”语义，则需要真正串行触发数据准备、证据分析、3D 融合、布孔和报告。
- 每个阶段按钮都应说明当前动作的真实后果，例如“开始下载 / 调用交付库”“运行所选证据”“进入 3D 融合”。
- 对已完成、失败、跳过、回退模拟状态使用统一状态标签。

### 2.4 项目空间

当前项目卡片展示名称、矿种、运行记录和角色，适合作为入口，但不够支撑实际项目管理。

建议增加：

- 最近运行时间。
- 当前运行阶段。
- 当前运行状态：运行中、失败、完成、待开始。
- AOI 文件状态：已上传、样例、缺失。
- 最近失败原因摘要。
- 创建人、更新时间、项目角色。
- 删除入口改为更多菜单，降低误触风险。

### 2.5 交互一致性

当前混用了 antd 组件和大量自绘 `button`、`span`、`b` 可点击元素，交互状态和可访问性不统一。

建议：

- 抽象统一的 Button、IconButton、Chip、StatusBadge、MetricRow、InspectorSection。
- 所有可点击元素使用明确的 button 语义或补足键盘交互。
- 统一 hover、focus、disabled、loading 状态。
- 统一图标体系，避免符号按钮含义不清。

### 2.6 安全与维护

顶部 crumb 当前使用 HTML 字符串渲染，后续如果项目名或阶段名进入 HTML 字符串，存在 XSS 风险。

建议：

- 将 crumb 改为结构化 props，例如 `projectName`、`mineralLabel`、`stageTitle`。
- 使用 JSX 渲染，不使用 `dangerouslySetInnerHTML`。
- 将大量内联样式收敛为组件样式或 CSS class，降低维护成本。

## 3. 证据链体系优化

### 3.1 核心原则

地质勘探找矿系统的核心不是“展示一个预测结果”，而是建立可复核的推理链。前端应支持用户从任意结论反查：

- 使用了哪些输入数据。
- 调用了哪些服务、算法和参数。
- 生成了哪些中间产物。
- 哪些证据支持该结论。
- 哪些证据缺失、失败、跳过或冲突。
- 当前结论的不确定性来自哪里。
- 最终报告是否能被审计和复现。

### 3.2 靶点详情升级为证据链面板

当前靶点详情主要展示评分、深度、不确定性和坐标。建议升级为“证据摘要 + 可展开证据链”。

靶点详情应包含：

- 基础信息：靶点编号、评分、深度、坐标、不确定性。
- 证据贡献：蚀变、构造、物探、化探、形变等证据对该靶点的贡献等级。
- 空间解释：是否位于异常区、距主断裂距离、异常重合度、化探点位关系。
- 反证与风险：缺失数据、低覆盖数据、与结论冲突的数据。
- 来源追踪：服务名、task_id、产物路径、参数版本、运行时间。

建议贡献等级使用统一枚举：

- 强支撑
- 中支撑
- 弱支撑
- 冲突
- 缺失
- 不参与

### 3.3 阶段级输入-处理-输出链路

每个阶段都应有明确的链路信息，而不是只显示百分比。

建议每个阶段展示：

- 输入：文件、AOI、数据源、上游产物、参数。
- 处理：调用服务、算法、模型版本、运行配置。
- 输出：栅格、矢量、JSON、图表、靶点、钻孔建议、报告。
- 质量：覆盖率、分辨率、时间范围、成功/失败/跳过、置信度。
- 审计：trace_id、task_id、开始时间、完成时间、真实/回退状态。

各阶段示例：

- 数据准备：遥感影像、DEM、物探、化探、知识库输入与下载/交付库来源。
- 证据分析：每个证据服务消费的数据源、生成的异常层、是否有可视栅格。
- 3D 建模：使用哪些证据层、融合方法、模型统计、靶点生成依据。
- AI 布孔：靶点选择、孔距约束、深度约束、布孔优先级、闭环反馈。
- 报告：纳入的证据、结论、反证、不确定性和审计信息。

### 3.4 证据层升级为证据矩阵

当前证据层 HUD 主要是图层开关和透明度控制。建议增加证据矩阵，让专家能快速判断证据质量和结论支撑情况。

证据矩阵字段建议：

| 证据 | 状态 | 数据质量 | AOI 覆盖 | 与靶点关系 | 结论贡献 |
| --- | --- | --- | --- | --- | --- |
| 蚀变 | 完成 | 高 | 92% | 位于强异常边界 | 强支撑 |
| 构造 | 完成 | 中 | 100% | 距主断裂 350m | 强支撑 |
| 物探 | 完成 | 中 | 80% | 异常重合 | 中支撑 |
| 化探 | 无栅格 | 低 | 点位稀疏 | 局部异常 | 弱支撑 |
| 形变 | 缺失 | - | - | 无直接支撑 | 不参与 |

证据矩阵不应替代图层控制，而应与图层控制并存：

- 图层控制解决“看什么”。
- 证据矩阵解决“为什么可信”。

### 3.5 明确展示冲突、缺失和回退

严谨的证据链不能只展示支持结论的证据，也必须展示反证和限制条件。

界面应主动暴露：

- 哪些证据没有覆盖当前 AOI。
- 哪些证据失败、跳过或没有可视化栅格。
- 哪些证据与高评分靶点不一致。
- 当前结论是否依赖 fallback/mock/示例数据。
- 是否存在“单一证据强、综合证据弱”的情况。
- 是否存在关键证据缺失但仍继续建模的情况。

建议使用统一标签：

- 真实结果
- 回退模拟
- 无覆盖
- 无栅格
- 失败
- 已跳过
- 证据冲突
- 低置信

### 3.6 常驻 Inspector

建议将右侧 HUD 升级为常驻 Inspector，并提供多个 tab：

- 操作：运行、重试、跳过、进入下一阶段。
- 证据链：当前阶段或当前靶点的证据来源、贡献、缺失、冲突。
- 产物：栅格、矢量、JSON、图表、报告、日志入口。
- 审计：trace_id、task_id、服务名、参数、运行时间、真实/回退状态。

这样可以同时保留沉浸式画布，又满足专业用户的审查需求。

### 3.7 报告阶段输出可审计证据链

报告不应只是“3D 预测 + 布孔章”的摘要，而应输出完整可审计证据链。

报告建议包含：

- 项目信息、AOI、矿种、trace_id。
- 数据来源清单。
- 各证据层处理过程。
- 证据质量评价。
- 综合解释逻辑。
- 靶区/靶点推荐理由。
- 反证与不确定性。
- 钻孔建议及依据。
- 服务 task_id、参数版本、产物路径、生成时间。

## 4. 建议实施顺序

### 第一阶段：低成本可信度增强

- 修改误导性按钮文案。
- 补充真实/回退/mock/失败/跳过状态标签。
- 靶点详情增加证据贡献列表。
- 证据阶段增加简版证据矩阵。
- 每个阶段 HUD 显示输入、输出和 task_id。
- 移除 crumb HTML 字符串渲染。

### 第二阶段：证据链主界面

- 将右侧 HUD 改造为 Inspector tabs。
- 建立前端 evidence graph 数据结构。
- 支持从靶点反查到证据层、服务任务和产物。
- 增加冲突证据和缺失证据展示。
- 项目空间展示当前阶段、最近运行和失败摘要。

### 第三阶段：专业化审查与报告

- 报告生成纳入完整证据链。
- 增加证据质量评价和覆盖率。
- 增加靶点-证据空间关系分析。
- 引入更真实的地图/剖面/三维表达。
- 建立截图级视觉回归和关键流程验收。

## 5. 验收标准

### 设计与可用性

- 在 `1440x900`、`1280x720`、`1024x768` 和窄屏下，顶部栏、HUD、dock、证据链面板不互相遮挡。
- 所有核心按钮文案能准确表达实际行为。
- 项目空间能快速判断项目状态、最近运行和是否存在失败。
- 交互控件具备统一的 hover、focus、disabled、loading 状态。

### 证据链

- 点击任一靶点，可以看到支撑证据、冲突证据和缺失证据。
- 每个证据都能显示状态、质量、覆盖率、来源和贡献等级。
- 每个阶段都能显示输入、处理、输出和 task_id。
- 服务失败、无栅格、无覆盖、跳过、fallback/mock 都有明确 UI 表达。
- 报告内容包含 trace_id、task_id、输入数据、处理过程、产物、结论依据和不确定性说明。

## 6. 关键默认假设

- 文档作为前端后续改造的设计和实施依据，不直接改变现有业务逻辑。
- 短期优先补充可解释性和状态表达，不强制立即替换 2D/3D 技术栈。
- 证据链 UI 应服务专业审查，不应只作为装饰信息。
- 演示数据、真实数据和回退模拟必须在界面上明确区分。
